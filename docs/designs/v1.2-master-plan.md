# Solo Squad v1.2 Master Plan

> From marketplace to operating system. The jump from "collection of skills" to "coordinated agency platform."

## Problem Statement

Solo Squad v1.0 shipped 6 squads with 36 skills, 11 agents, and 18 commands. It works — but each squad operates in isolation. Agent definitions are thin (role + tool permissions), skills don't learn from each other, there's no way to coordinate squads on a single client engagement, and the skill library has clear gaps in growth marketing, DevOps, database work, analytics, and legal.

Agency Agents (msitarzewski/agency-agents) demonstrates what depth looks like: 178 agents with rich behavioral directives, decision frameworks, and domain expertise. Their NEXUS orchestration system shows what cross-agent coordination can be. Solo Squad needs to absorb these lessons without abandoning its plugin architecture advantage.

## Vision

v1.2 transforms Solo Squad from "a collection of specialist skills" into "a coordinated agency operating system." After v1.2:

1. Every agent has a defined personality, decision framework, and quality standard
2. 18 existing skills are sharpened with insights from Agency Agents
3. 18 new skills fill the gaps in growth, DevOps, analytics, legal, and paid social
4. NEXUS orchestration coordinates all 6 squads on a single client engagement
5. Persistent client context carries intelligence across sessions

## Scope

### In Scope

- Agent enrichment standard applied to all agents (existing + new)
- 18 improvements to existing skills/agents
- 18 new skills + 1 new agent
- V1.1 roadmap: MEDDPICC command, design system templates, smarter hooks

## Design Decisions

### D1: Agent Enrichment Standard

Every agent (existing and new) follows this template:

```markdown
---
name: "{Agent Name}"
description: "{One sentence — when to dispatch this agent}"
model: inherit
---

# {Agent Name}

{One paragraph: who you are and what you do. Written in second person.}

## Expertise

{3-7 bullet points of specific domain knowledge. Not generic — name methodologies, frameworks, tools.}

## Decision Framework

{How this agent makes tradeoffs. What it optimizes for. What it sacrifices.}

## Behavioral Directives

- {Specific behavioral rules — e.g., "Never guess. If evidence is insufficient, say so."}
- {Communication style — e.g., "Lead with the recommendation, then the reasoning."}
- {Quality gate — e.g., "Only report findings at confidence 8/10 or higher."}

## Deliverables

{What this agent produces. Format, structure, length expectations.}
```

**What we're adding vs current:** Decision Framework, Behavioral Directives, and structured Deliverables sections. These come from AA's pattern of rich behavioral definition. We're NOT adding personality fluff (emoji, color, vibe) — only directives that improve output quality.

**Why:** Current agents have 1-2 paragraphs of role description. AA agents have 200+ lines of behavioral precision. The gap shows in output quality — a thin agent definition produces generic responses.

### D2: NEXUS Lives in project-squad

NEXUS orchestration goes into project-squad, not a separate plugin.

**Rationale:** Orchestration IS project management. Studio Producer already coordinates multi-project work. NEXUS extends this to multi-squad coordination. Adding a 7th plugin increases install friction for no architectural benefit.

**Implementation:**
- New agent: `Orchestrator` in project-squad/agents/
- New skills: `nexus-full`, `nexus-sprint`, `nexus-micro` in project-squad/skills/
- New command: `/orchestrate` in project-squad/commands/
- Handoff templates and quality gates embedded in skill files (not separate docs)

### D3: Client Context via Markdown Files

Persistent client context uses a structured `.md` file per client, managed by a project-squad skill.

**Structure:**
```
clients/
  {client-slug}/
    context.md      # Company info, contacts, brand ref, preferences
    history.md      # Engagement history, decisions, learnings
    active.md       # Current projects, status, blockers
```

**Location:** `solo-squad/clients/` — lives alongside the plugin, not inside project repos.

**Rationale:** Simple, version-controllable, works without external dependencies. Keeping it in the plugin directory means client data travels with the Solo Squad install. QMD integration is a future enhancement — the .md structure is forward-compatible.

### D4: New Skills vs New Agents

17 of 18 new additions are **skills** (process-driven workflows). Only 1 is a new **agent**:

- **Sales Engineer** (sales-squad agent) — needs a distinct technical pre-sales persona with specific tool permissions for demo engineering and POC scoping

Everything else is a skill because the value is in the structured process, not in a dispatchable persona.

### D5: Smarter Hooks

Current hooks fire generic reminders after every tool use. v1.2 hooks are context-aware using shell scripts that receive full tool input JSON on stdin, run conditional logic, and return `additionalContext` only when relevant.

**Implementation pattern:** Each squad gets a `hooks/smart-reminder.sh` script. The `hooks.json` uses `matcher` to filter by tool name. The script reads `tool_input.file_path` via `jq`, checks file extension and related file existence, and conditionally returns context.

| Squad | Matcher | Condition | Fires when |
|-------|---------|-----------|------------|
| Engineering | `Write\|Edit` | `file_path` is `.ts/.tsx/.js/.jsx`, no corresponding `.test.*` file exists | Editing implementation without test coverage |
| Marketing | `Write\|Edit` | `file_path` is `.html/.astro/.svelte/.tsx` AND contains `href=` or `action=` | Editing pages/templates with links or forms |
| Design | `Write\|Edit` | `file_path` is `.css/.scss/.svg` OR contains image references | Editing style/visual files |
| Sales | `Write\|Edit` | `file_path` is inside `proposals/` or `outreach/` directories | Writing client-facing sales documents |
| Project | `Bash` | Command contains `git push`, `deploy`, or `vercel` | Deployment or significant operations |
| Security | `Bash` | Command matches destructive pattern AND not in session whitelist | Destructive commands (rm -rf, DROP, force-push) |

### Phase 7: NEXUS Orchestration

**Goal:** Build cross-squad coordination into project-squad. This is the signature v1.2 feature.

#### Architecture

NEXUS adds to project-squad:
- 1 new agent: **Orchestrator**
- 3 new skills: `nexus-full`, `nexus-sprint`, `nexus-micro`
- 1 new command: `/orchestrate`

#### The Orchestrator Agent

```
agents/orchestrator.md
```

Pipeline manager that coordinates multi-squad workflows. Decides which skills to invoke from which squads, enforces quality gates between phases, manages handoff context, and reports progress.

**Decision Framework:** Minimize calendar time while maintaining quality gates. Parallelize independent work streams. Never skip a quality gate — delay is acceptable, shipping broken work is not.

**Behavioral Directives:**
- Always confirm scope and mode (Full/Sprint/Micro) before starting
- Display estimated cost/time at kickoff
- Pause for human approval between major phases
- Log every skill invocation with input/output summary for auditability

#### Three Operating Modes

**NEXUS-Full** (4-12 weeks, comprehensive)
```
Phase 0: Discovery     → /brainstorm + /ux-research + /seo-audit + industry research
Phase 1: Strategy      → /plan + /design-system + /product-strategy + /client-kickoff
Phase 2: Foundation    → /devops-pipeline + /workflow-mapping + design tokens
Phase 3: Build         → /build + /review cycles (Dev-QA loops until quality gate passes)
Phase 4: Hardening     → /cso + /benchmark + /qa + /legal-compliance
Phase 5: Launch        → /ship + /social-strategy + /paid-social + /content-plan + /ad-creative
Phase 6: Operate       → /status-report + /analytics-reporting + /experiment-track + /retro
```

**NEXUS-Sprint** (2-6 weeks, compressed)
```
Discovery + Strategy (combined)  → /brainstorm + /plan + /client-kickoff
Build + Review                   → /build + /review (concurrent QA)
Harden + Ship                    → /cso + /qa + /ship
Launch (optional)                → /content-plan + /social-strategy
```

**NEXUS-Micro** (1-5 days, rapid)
```
Scope → Build → Ship (single stream, minimal ceremony)
```

#### Quality Gates

Each phase transition requires passing a quality gate:

| Gate | Criteria |
|------|----------|
| Discovery → Strategy | Problem statement validated, success criteria defined, risks identified |
| Strategy → Foundation | Architecture approved, plan reviewed, dependencies mapped |
| Foundation → Build | CI/CD green, dev environment works, design tokens available |
| Build → Hardening | All tasks complete, tests pass, coverage meets threshold |
| Hardening → Launch | Security audit passed, performance budgets met, QA signed off |
| Launch → Operate | Deployment successful, monitoring active, runbook exists |

#### Handoff Protocol

When one skill completes and hands off to the next, the handoff includes:

```markdown
## Handoff: {Source Skill} → {Target Skill}

### Context
{What was done and why}

### Deliverables
{List of artifacts produced with file paths}

### Decisions Made
{Key decisions and their rationale}

### Open Items
{Anything unresolved that the next skill needs to address}

### Quality Gate Status
{Pass/Fail with evidence}
```

#### Scenario Runbooks

Pre-configured pipeline configurations for common scenarios:

1. **Client Website Rebuild** — Full mode, all squads
2. **New Feature Sprint** — Sprint mode, engineering + design
3. **Marketing Campaign** — Sprint mode, marketing + design + project
4. **Production Incident** — Micro mode, engineering + security

**Deliverable:** 1 agent, 3 skills, 1 command, 4 scenario runbook definitions embedded in skills.

### Phase 8: Client Context System

**Goal:** Persistent client intelligence that carries across sessions.

#### New Skill: Client Context (project-squad)

**File:** `skills/client-context/SKILL.md`

**Commands:**
- `/client-context init {client-name}` — create a new client context directory
- `/client-context load {client-name}` — load client context into session
- `/client-context save` — update context files with session learnings
- `/client-context list` — list all client contexts

**Directory Structure (inside solo-squad plugin root):**

```
clients/
  acme-widgets/
    context.md        # Immutable facts: company info, industry, contacts, tech stack
    brand.md          # Brand guidelines reference, voice/tone, visual identity
    history.md        # Engagement history: what we've done, decisions made, learnings
    active.md         # Current projects: status, blockers, next steps
```

**Context Loading:** When `/client-context load` runs, the skill reads all 4 files from the plugin's `clients/` directory and injects a summary into the session. All subsequent skill invocations benefit from this context — the SEO analyst knows the client's industry, the proposal writer knows past engagements, the orchestrator knows current project state.

**Context Saving:** After significant work (audit, sprint, campaign), the skill prompts to update the relevant context files with new information.

**Deliverable:** 1 new skill, client directory structure definition.
